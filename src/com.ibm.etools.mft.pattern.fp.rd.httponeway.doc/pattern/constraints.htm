Constraints on the use of the Record Distribution to HTTP: one-way pattern

A number of constraints apply when you use this pattern.

You can use the Record Distribution to HTTP: one-way pattern only if all of the following conditions are met:

  • It must be possible to read the input file as comprising a number of individual records.
    Files can be processed if they contain fixed-length records, variable-length records that are separated by a well-known delimiter, or variable-length records that can be extracted by a parser; a technique referred to as Parsed Record Sequence. You can use this technique to read a file that contains multiple XML documents, and return each document as a record. You cannot use this pattern to extract embedded records; for example, records within a single XML document.

  • The integration node must be able to delete the source file.
    Source files are removed from their original location and moved to the integration node; they are not read in place.

  • The business function must be compatible with the redistribution of records after a system failure.
    This pattern distributes the records reliably, but does not guard against duplicate records after a system failure.

  • The business function must be compatible with the redistribution of records after a failure to read a record from the file, or to route a record to a destination.
    In this case, the whole file is moved to the backout directory. You must arrange for the file to be resubmitted after the error is corrected. When a file is resubmitted, all the records in the file are reprocessed, which might mean that the HTTP web service receives duplicate records.
